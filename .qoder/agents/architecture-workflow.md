## 九大阶段详解

### 第零阶段：需求深度沟通（Requirements Deep Dive）- 可多轮

**目标**：与用户深入沟通，产出高质量、无歧义的需求文档

**参与者**：
- 产品经理 (product-manager)

**流程**：

```
用户提出需求
    │
    v
主流程调用产品经理
    │
    ├─ 需求解构（核心目标/目标用户/使用场景/痛点/成功标准）
    ├─ 竞品分析（按需联网调研，对比3-5个竞品）
    ├─ 可行性预判（产品层面：用户价值/市场需求/逻辑一致性）
    ├─ 创新性分析（功能延伸/场景延伸/体验优化/数据价值）
    └─ 风险识别（不合理需求标记 + 替代方案）
    │
    v
输出：需求分析报告 + 确认清单
    │
    v
主流程将报告和确认清单展示给用户
    │
    ├─ 用户确认需求 → 进入架构设计（阶段1）
    ├─ 用户调整需求 → 再次调用产品经理（携带用户反馈）
    └─ 用户取消项目 → 流程结束
```

**重要机制**：
- 产品经理必须主动识别不合理需求，给出专业解释和替代方案
- 创新建议必须标注优先级（P0核心/P1增强/P2远期），避免功能膨胀
- 用户是专业人士，沟通时应直接、平等，不回避问题

**输出**：
- `docs/requirements.md`（产品需求分析报告）

**注意**：产品经理不涉及技术实现，技术可行性由后续架构师判断

---

### 架构师与产品经理的反馈机制

**当架构师在阶段1-4中发现功能技术上不可行时**：

```
架构师标记不可行功能
    │
    v
主流程收集架构师反馈
    │
    v
主流程调用产品经理（场景二：架构反馈沟通）
    │
    ├─ 理解技术不可行原因
    ├─ 转化为用户能理解的语言
    ├─ 提出替代方案
    └─ 输出需求调整说明
    │
    v
主流程展示调整方案给用户
    │
    ├─ 用户接受调整 → 更新 requirements.md → 继续架构设计
    ├─ 用户提出其他方案 → 再次调用产品经理
    └─ 用户坚持原方案 → 记录为已知风险，继续推进
```

### 第一阶段：多平行架构生成（Diversity）

**目标**：生成风格迥异的三个架构方案

**参与者**：
- 架构师 A (architect-a) → 输出 ARCH_DESIGN_v1.md
- 架构师 B (architect-b) → 输出 ARCH_DESIGN_v2.md
- 架构师 C (architect-c) → 输出 ARCH_DESIGN_v3.md

**输入**：
- 产品需求文档（来自产品经理）

**输出**：
- 三份风格迥异的架构设计文档

**执行方式**：并行调用三个架构师

> 如在本阶段需要对关键组件/技术路线进行选型，架构师可先调用 `solution-researcher` 进行「现成方案/轮子」调研，再在各自方案中引用或对比这些调研结论。

---

### 第二阶段：红蓝对抗辩论（Selection）- 最多5轮

**目标**：暴露每个方案的盲点和缺陷

**参与者**：
- 批判性审核员 (critical-reviewer) - 主导辩论
- 高级程序员 (senior-developer) - 从实现角度提问
- 三个架构师 - 为自己的方案辩护

**重要：5轮底线机制仅适用于辩论阶段**

```
辩论轮次 1: 批判 → 辩护 → 评估 → [共识达成/继续]
辩论轮次 2: 批判 → 辩护 → 评估 → [共识达成/继续]
辩论轮次 3: 批判 → 辩护 → 评估 → [共识达成/继续]
辩论轮次 4: 批判 → 辩护 → 评估 → [共识达成/继续]
辩论轮次 5: 批判 → 辩护 → 强制进入整合阶段
```

**每轮辩论流程**：
1. 批判性审核员针对当前方案提出尖锐问题
2. 高级程序员从实现角度提出质疑
3. 架构师为自己的方案辩护
4. 评估是否达成共识
   - 共识达成 → 进入整合阶段
   - 未达成共识 → 继续下一轮辩论
   - 第5轮 → 强制进入整合阶段

**输出**：
- 架构方案批判报告
- 辩论轮次记录

#### 辩论协议（共识定义与记录方式）

为保证红蓝对抗辩论可以在有限轮次内**收敛到可执行的共识方案**，本阶段遵循以下协议：

1. **输入与产物**
   - 第 1 轮输入：`docs/arch/ARCH_DESIGN_v1.md`、`ARCH_DESIGN_v2.md`、`ARCH_DESIGN_v3.md` 与需求文档 `docs/requirements.md`。
   - 第 N (N>1) 轮额外输入：上一轮的辩论记录 `docs/debates/debate_round_{N-1}.md` 与最新的架构师回应。
   - 每一轮结束时，必须产出：
     - `docs/debates/debate_round_N.md`：本轮详细问答与未解决问题；
     - 如为最后一轮或达到共识，更新/生成 `docs/arch/CRITIQUE_REPORT.md` 总结各方案优劣与推荐结论。

2. **每轮固定流程**
   1. 批判性审核员阅读当前输入后，生成当轮问题与场景挑战，写入 `debate_round_N.md`。
   2. 三位架构师分别针对本轮问题为各自方案辩护，追加到同一轮文档中。
   3. 高级程序员从实现视角补充质疑与现实约束（成本、复杂度、维护性等）。
   4. 批判性审核员对本轮进行小结，标记：
      - 已解决的问题；
      - 仍然开放的致命/重要问题；
      - 候选「主方案」与「备选方案」。

3. **共识达成条件（满足以下全部视为达成共识）**
   - 在最新一轮的总结中，**不存在未解决的致命问题**（安全性、可行性、关键性能等方面）；
   - 高级程序员认为在工程实现上**整体可行**（可以用伪代码预演验证）；
   - 批判性审核员在 `CRITIQUE_REPORT.md` 中明确：
     - 至少一个**推荐主方案**（可为混合/杂交方向，而非单一原始方案）；
     - 其他方案标记为「备选/局部可借鉴」；
     - 各关键取舍（性能 vs 可维护性、开发效率 vs 风险等）已有清晰记录；
   - 三位架构师中**至少一位明确认可**推荐主方案为可接受折中，其他人如仍有分歧，也需将分歧点显式写入报告作为「已知权衡」而非阻塞点。

4. **轮次与强制收敛**
   - 若在第 1~4 轮任一轮满足上面的共识条件，则提前结束辩论，进入整合阶段；
   - 若进行到第 5 轮仍有分歧：
     - 批判性审核员必须在 `CRITIQUE_REPORT.md` 中给出**强制推荐主方案**（或明确的组合方向）；
     - 将未达成一致的观点全部作为「已知权衡/已知风险」记录，由后续整合者与用户确认环节感知；
     - 辩论阶段在第 5 轮结束后**不再继续扩展轮次**。

### 第三阶段：架构杂交与变异（Crossover & Mutation）

**目标**：整合各方长处，生成最终演化版架构

**参与者**：
- 整合者 (synthesizer)

**流程**：
1. 解构三个方案的设计决策
2. 识别每个决策的优劣
3. 杂交整合：选择最优决策组合
4. 变异创新：针对共性难题引入新策略
5. 生成演化版架构

**输出**：
- Evolved_ARCH.md（演化版架构）

---

### 第四阶段：程序员预演与反馈（Validation）

**目标**：验证架构在工程上的可行性

**参与者**：
- 高级程序员 (senior-developer)

**流程**：
1. 编写核心模块伪代码
2. 验证模块间接口匹配
3. 发现潜在的逻辑冲突
4. 给出预演结论

**结果处理**：
- 通过 → 进入用户确认环节
- 不通过 → 打回第三阶段重新整合

---

### 第五阶段：用户确认（Confirmation）- 关键节点

**目标**：确保用户认可架构设计后再开始开发

**重要性**：这是整个流程的关键决策点，避免在用户不满意架构的情况下浪费开发资源。

#### 确认流程

```
架构预演通过
    │
    ├─ 整理架构摘要
    │   ├─ 核心设计理念
    │   ├─ 技术选型
    │   ├─ 模块划分
    │   ├─ 接口概览
    │   └─ 风险与权衡
    │
    ├─ 展示给用户
    │   └─ 使用 AskUserQuestion 工具
    │
    └─ 等待用户决策
        ├─ 确认开发 → 进入代码实现
        ├─ 修改架构 → 返回整合者
        └─ 取消项目 → 流程结束
```

```markdown
3. 取消项目 - 暂不继续
```

#### 用户选项处理

| 用户选择 | 处理方式 |
|---------|---------|
| 确认开发 | 进入阶段6（代码实现） |
| 修改架构 | 收集用户修改意见，返回整合者重新演化 |
| 取消项目 | 流程结束，记录项目状态 |

#### 修改架构流程

如果用户选择修改架构：

```
用户提出修改意见
    │
    ├─ 记录修改需求
    │
    ├─ 分析影响范围
    │   ├─ 影响哪些模块
    │   ├─ 影响哪些接口
    │   └─ 影响哪些技术选型
    │
    └─ 返回整合者
        │
        └─ 重新演化架构 → 再次用户确认
```

### 第六阶段：代码实现（Implementation）- 分而治之

**目标**：将架构设计转化为可执行代码

**前提条件**：用户已确认架构设计

> 如在本阶段发现某个功能模块实现复杂或社区已有成熟方案（例如认证、文件上传、搜索、工作流、ASR/OCR 等），高级程序员可在拆解任务前调用 `solution-researcher` 先行调研可复用的开源项目或库，再将选用方案写入任务分发单。

#### 任务分发者：高级程序员

**职责**：
1. 分析架构设计文档
2. 拆解为子任务
3. 为每个子任务提供上下文
4. 分发给对应的子程序员
5. 汇总各子程序员输出，并整理为 code-integrator 的输入（模块清单 + 路径 + 任务分发记录 + 重要依赖关系）

**任务分发模板**：

```markdown
# 任务分发单

## 任务ID：TASK-001

### 基本信息
- 任务类型：[前端/后端/数据库/AI]
- 优先级：[高/中/低]
- 依赖任务：[无/TASK-XXX]

### 任务描述
[具体的任务描述]

### 架构上下文
- 相关架构设计：[引用 Evolved_ARCH.md 相关章节]
- 接口定义：[需要实现的接口]
- 数据结构：[相关的数据模型]

### 依赖关系
- 上游依赖：[其他模块提供的接口]
- 下游影响：[本模块需要提供的接口]

### 约束条件
- 技术栈：[指定的技术]
- 性能要求：[如响应时间、并发量]
- 安全要求：[认证、授权、数据加密]

### 验收标准
- [ ] 功能完整
- [ ] 接口符合定义
- [ ] 单元测试通过
```

#### 执行者：子程序员

| 子程序员 | 职责 | 输出 |
|---------|------|------|
| frontend-developer | 前端UI、交互逻辑、页面组件 | 前端代码 |
| backend-developer | 后端API、业务逻辑、服务层 | 后端代码 |
| database-developer | 数据库设计、SQL优化、数据迁移 | 数据库代码 |
| ai-developer | AI接口调用、模型集成、Prompt设计 | AI相关代码 |

#### 代码集成者：code-integrator

**职责**：
1. 在子程序员完成各自任务后，接收由高级程序员整理的模块清单与任务分发记录；
2. 对前端、后端、数据库、AI 等模块执行接口与数据模型的一致性检查；
3. 识别并解决模块间的依赖冲突与配置/版本冲突；
4. 执行基础集成验证（例如前端→后端→数据库/AI 的接口调用是否能在开发环境中打通）；
5. 生成 `docs/reviews/` 或项目内约定位置的**代码集成报告**，并将集成后的代码与报告交回高级程序员，用于后续构建/跑通与质量保障。

---

### 第七阶段：代码审查（Review）

**目标**：确保代码质量，防止屎山代码

**参与者**：
- 代码审查员 (code-reviewer)

**六大审查维度**：

| 维度 | 检查内容 | 否决级别 |
|------|---------|---------|
| 冗余性 | 重复代码、无用代码、过度封装 | 中 |
| 安全性 | SQL注入、XSS、认证绕过、敏感数据 | **致命** |
| 合理性 | 函数长度、嵌套层级、复杂度 | 中 |
| 文档合规 | 是否按文档实现、有无越权功能 | **高** |
| 可维护性 | 可读性、可测试性、屎山代码预警 | 高 |
| 性能 | N+1查询、内存泄漏、性能瓶颈 | 高 |

**结果处理**：
- ✅ 通过 → 进入产品测试
- ⚠️ 有条件通过 → 修复后进入产品测试
- ❌ 否决 → 打回高级程序员重新实现

---

### 第八阶段：产品测试（Testing）- 最终验收

**目标**：验证功能完整性，发现Bug

**参与者**：
- 产品测试员 (product-tester)

**测试维度**：

| 维度 | 检查内容 | 通过标准 |
|------|---------|---------|
| 功能完整性 | 所有需求功能是否实现 | 100%覆盖 |
| 功能正确性 | 功能是否按预期工作 | 符合需求文档 |
| Bug检测 | 是否存在运行时错误 | 无阻塞性Bug |
| 边界测试 | 边界条件是否处理 | 无异常 |
| 集成测试 | 模块间集成是否正常 | 接口调用正常 |
| 用户体验 | 是否符合产品预期 | 符合产品经理定义 |

**结果处理**：
- ✅ 通过 → 项目完成
- ❌ 不通过 → 打回高级程序员（带Bug报告）

## 持续迭代机制

### 整体流程不设上限

```
需求沟通（可多轮）
    ↓
架构设计（辩论最多5轮）
    ↓
用户确认 ←─────┐
    ↓          │
代码实现        │
    ↓          │
代码审查        │
    ↓          │
产品测试        │
    ↓          │
┌─ 通过？       │
│   │          │
│   否 ────────┘
│   │
│   是
│   ↓
└─ 项目完成
```

### 迭代场景

| 场景 | 触发条件 | 返回阶段 | 携带信息 |
|------|---------|---------|---------|
| 需求不明确 | 产品经理需要用户澄清 | 阶段0（产品经理） | 确认清单 |
| 架构发现不可行 | 架构师标记功能技术不可行 | 阶段0（产品经理） | 架构师反馈 |
| 用户要求修改 | 用户对架构不满意 | 整合者 | 修改意见 |
| 代码审查否决 | 安全漏洞/文档违规 | 高级程序员 | 审查报告 |
| 产品测试失败 | Bug/功能缺失 | 高级程序员 | Bug报告 |
| 架构预演失败 | 逻辑冲突 | 整合者 | 预演报告 |

## 完整调用流程

```
# ========== 需求沟通阶段 ==========
0. product-manager → 需求分析（需求解构/竞品分析/可行性预判/创新分析）
   - 输出需求分析报告 + 确认清单
   - 用户确认需求 → 进入架构设计
   - 用户调整需求 → 再次调用 product-manager
   - 取消项目 → 流程结束

# ========== 架构设计阶段 ==========
1. architect-a/b/c 并行 → 三方案（输入：requirements.md）
2. critical-reviewer + senior-developer → 辩论（最多5轮）
   - 架构师发现功能不可行 → 调用 product-manager 进行需求调整 → 用户确认
3. synthesizer → 整合方案
4. senior-developer → 伪代码预演
   - 不通过 → 返回步骤3
   - 通过 → 进入用户确认

# ========== 用户确认阶段 ==========
5. 展示架构给用户确认
   - 确认开发 → 进入代码实现
   - 修改架构 → 收集意见，返回步骤3
   - 取消项目 → 流程结束

# ========== 代码实现阶段 ==========
6. senior-developer → 分析架构，拆解任务
7. senior-developer → 分发子任务（提供上下文）：
   - frontend-developer → 前端任务
   - backend-developer → 后端任务
   - database-developer → 数据库任务
   - ai-developer → AI任务
8. senior-developer → 汇总代码并整理为集成输入，交给 code-integrator
9. code-integrator → 执行接口/数据模型检查与集成验证，输出集成报告与集成后的代码
10. senior-developer → 基于集成结果执行整体构建/跑通，并将状态准备给代码审查员与产品测试员

# ========== 质量保障阶段 ==========
11. code-reviewer → 六维审查
    - 否决 → 返回步骤6（带审查报告）
    - 通过 → 进入测试

# ========== 产品测试阶段 ==========
12. product-tester → 功能测试
    - 不通过 → 返回步骤6（带Bug报告）
    - 通过 → 项目完成
```

## 关键机制总结

| 机制 | 说明 |
|------|------|
| 需求深度沟通 | 产品经理多轮分析，确保需求质量 |
| 架构-需求反馈 | 架构师发现不可行时，产品经理负责需求调整 |
| 5轮底线 | 仅限辩论阶段，防止无限争论 |
| 用户确认 | 需求确认 + 架构确认，两道关卡 |
| 持续迭代 | 整体流程不设上限，直到测试通过 |
| 分而治之 | 高级程序员分发任务，子程序员专注领域 |
| 上下文传递 | 任务分发时提供完整上下文和约束 |
| 六道防线 | 需求→架构→确认→实现→审查→测试，层层把关 |
