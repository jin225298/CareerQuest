## 1. 方案A质询（性能派）

### Q1: Go + Rust 混合架构，谁来维护？

文档声称"Go (主) + Rust (关键路径)"，但：

- **FFI 调用的开销**：每次 Go 调 Rust 都要穿越 CGO 边界，这个开销是多少？你量化过吗？
- **调试噩梦**：生产环境出问题时，是 Go 的问题还是 Rust 的问题？panic 和 error 如何统一处理？
- **团队成本**：你需要同时招聘 Go 和 Rust 工程师，薪资预算增加多少？
- **真实收益**：Rust 真的比优化后的 Go 快多少？有 benchmark 数据吗？

**尖锐追问**：如果只有 Go 工程师，这方案还能落地吗？如果不能，请给出 Plan B。

### Q2: 单机 50k WebSocket 连接，数据从哪来？

文档宣称"单机 50k 连接，内存 < 2GB"：

- **数学验算**：每个 WebSocket 连接至少需要读/写缓冲区、连接状态结构体。假设每个连接占 40KB（保守估计），50k 连接就需要 2GB。你的 "内存 < 2GB" 是怎么算出来的？
- **心跳开销**：30s 心跳间隔，50k 连接每秒要处理 1666 个心跳包，这不会成为瓶颈吗？
- **真实场景**：用户网络不稳定导致的断线重连，如何处理？重连风暴会不会拖垮服务器？
- **压测数据**：你有在真实网络环境下压测过吗？还是只是理论值？

**尖锐追问**：请提供真实的压测报告，包括 CPU、内存、网络 IO 的详细数据。

### Q3: 多级缓存的失效策略是什么？

文档设计了 L1 本地 < 1μs → L2 Redis < 5ms → L3 DB < 50ms：

- **缓存一致性**：用户属性（武力值、心情值）更新时，L1 如何失效？L1 和 L2 的同步延迟如何处理？
- **缓存穿透**：恶意请求不存在的数据，如何防护？
- **缓存雪崩**：大量缓存同时失效，如何避免数据库被打挂？
- **缓存击穿**：热点 Key 过期瞬间，如何防止并发请求穿透到数据库？

**尖锐追问**：请给出缓存失效的具体流程图和代码实现方案。

### Q4: 外部服务全挂了怎么办？

系统依赖：通义千问、讯飞ASR、阿里TTS、Seedance：

- **降级方案**：文档只说"切换备用服务商"，但通义千问挂了，切换到哪个？切换需要多久？切换期间的用户请求怎么处理？
- **成本失控**：多个服务商的 API 费用叠加，月成本会是多少？
- **数据一致性**：切换服务商后，之前的对话上下文还能用吗？

**尖锐追问**：请给出完整的故障演练脚本，包括每个外部服务的故障模拟和恢复流程。

### Q5: 边录边传，网络抖动怎么办？

文档设计"音频切片 100ms，WS流式上传"：

- **丢包处理**：某个音频切片丢失，后续如何补齐？音频会断裂吗？
- **乱序到达**：网络延迟导致切片乱序，ASR 能正确识别吗？
- **弱网环境**：用户在电梯、地铁等弱网环境下，体验会怎样？

**尖锐追问**：请给出弱网环境（丢包率 5%、延迟 200ms）下的用户体验评估。

## 2. 方案B质询（稳健派）

### Q1: 10 个容错点，谁来协调？

文档设计了 10 个容错点，每个都有熔断、重试、降级：

- **配置爆炸**：每个容错点都有独立的阈值配置，这些参数如何调优？谁来负责？
- **级联失败**：面试服务熔断 → 触发用户服务重试 → 触发 Redis 熔断 → 全局雪崩。这个链条如何预防？
- **熔断恢复**：熔断器从 Open 到 Half-Open，如果试探请求失败，会不会反复熔断？

**尖锐追问**：请给出容错点之间的协调机制，以及配置参数的调优方法论。

### Q2: DDD 落地，团队有这个能力吗？

文档采用 DDD 代码结构：domain/application/infrastructure/interfaces：

- **领域划分**：面试、像素、任务这些领域的边界在哪？跨领域操作如何处理？
- **过度设计**：对于 MVP 阶段，DDD 的复杂度是否必要？开发效率会降低多少？
- **团队培训**：团队成员是否熟悉 DDD？培训成本是多少？

**尖锐追问**：如果团队成员不熟悉 DDD，强行推行会导致什么后果？有没有渐进式落地方案？

### Q3: 安全防护的性能代价，用户能接受吗？

文档明确"可牺牲部分性能换取安全边界"：

- **延迟叠加**：HTTPS + 请求签名 + 数据加密，文档承认延迟增加 5-10ms。这是单次请求还是链路总延迟？
- **面试场景**：实时面试对延迟敏感，5-10ms 的额外延迟会影响用户体验吗？
- **成本考量**：加密解密的 CPU 开销，需要多少额外服务器？

**尖锐追问**：请给出安全防护前后的对比性能数据，包括 P50/P90/P99 延迟。

### Q4: 限流策略，用户会被误杀吗？

文档设计"单用户请求 60 req/min"：

- **面试场景**：一个 10 分钟的面试，如果每秒发送音频切片，就是 6000 次请求。60 req/min 的限制是不是太低了？
- **WebSocket 消息**：WebSocket 消息是否计入限流？如果是，一个面试就会触发限流。
- **误杀问题**：正常用户触发限流，如何申诉？

**尖锐追问**：请给出面试场景下的详细请求量估算，以及限流阈值的合理性论证。

### Q5: 熔断降级，用户体验如何？

文档设计"LLM 不可用 --> 返回预设回复"：

- **预设回复**：预设回复的内容是什么？用户会接受"AI 暂时不可用"的回复吗？
- **降级触发**：什么条件触发降级？错误率 > 50%？如果只是偶尔超时，也要降级吗？
- **恢复时间**：降级后多久能恢复？用户会流失多少？

**尖锐追问**：请给出降级状态下的完整用户体验流程，以及用户满意度预期。

## 3. 方案C质询（极速派）

### Q1: Supabase 挂了，整个产品就挂了？

文档完全依赖 Supabase：认证、数据库、存储：

- **SLA 风险**：Supabase 的 SLA 是多少？免费版有 SLA 保证吗？
- **数据主权**：用户数据存在 Supabase，合规性如何保证？
- **迁移成本**：如果 Supabase 停止服务或涨价，迁移需要多少成本？
- **单点故障**：Supabase 挂了，你的产品能做什么？只能等它恢复？

**尖锐追问**：请给出 Supabase 故障时的应急方案，以及数据迁移的成本估算。

### Q2: 外部服务依赖，谁来背 SLA？

系统依赖：通义千问、阿里云语音、Seedance、阿里云 OSS：

- **串联风险**：任何一个服务挂了，核心功能就不可用。这是你设计的"极速开发"，还是"极速翻车"？
- **成本黑洞**：用户增长 → API 调用量增长 → 成本线性增长。月成本 ¥800-3000 是多少用户量下的估算？
- **API 限流**：通义千问、阿里云都有 API 调用限制，用户多了怎么办？

**尖锐追问**：请给出用户增长到 1 万、10 万、100 万时的成本估算。

### Q3: Node.js 单线程，面试并发怎么做？

文档使用 Node.js 作为后端：

- **CPU 密集型**：音频处理（编码、解码、格式转换）是 CPU 密集型任务。Node.js 单线程如何处理？
- **并发面试**：100 个用户同时面试，WebSocket + 音频处理 + LLM 调用，Node.js 能扛住吗？
- **GC 停顿**：大量音频数据处理，GC 停顿会影响实时性吗？

**尖锐追问**：请给出 Node.js 后端的并发能力压测数据，以及与 Go/Java 的对比。

### Q4: 技术债务，什么时候还？

文档明确列出技术债务：无单元测试、无监控告警、无数据备份：

- **债务累积**：MVP 上线后，是新功能优先还是还债务优先？
- **质量崩塌**：没有测试和监控，线上 Bug 如何发现和定位？
- **数据丢失**：没有备份，数据库挂了怎么办？

**尖锐追问**：请给出技术债务的偿还计划，包括具体时间点和负责人。

### Q5: MVP 4 周，质量如何保证？

文档计划 MVP 4 周/132 人时：

- **工时估算**：132 人时 = 3.5 周 * 1 人。一个人能完成认证、像素生成、面试核心、评分系统？
- **测试时间**：测试和 Bug 修复的时间算进去了吗？文档说 W4 集成测试 16h，这够吗？
- **真实经验**：类似项目的历史数据是什么？4 周是乐观估计还是保守估计？

**尖锐追问**：请给出详细的工作分解结构（WBS），包括每个任务的负责人、前置依赖、风险点。

## 6. 共性盲点

### 盲点 1：外部服务的成本模型缺失

三个方案都依赖大量外部服务（LLM、ASR、TTS、像素生成），但：

- **成本预测**：没有给出用户增长后的成本曲线。
- **成本控制**：没有给出 API 调用量的监控和预警机制。
- **成本优化**：没有考虑缓存、批处理等成本优化手段。

### 盲点 2：数据安全与合规

三个方案都没有详细说明：

- **数据留存**：用户语音、面试记录保留多久？如何删除？

### 盲点 3：用户体验降级方案

三个方案都假设外部服务正常，但：

- **降级体验**：外部服务不可用时，用户看到什么？

- **补偿机制**：因服务故障导致的用户损失，如何补偿？

### 盲点 4：可观测性

三个方案都没有完整的可观测性设计：

- **日志**：日志格式、日志量、日志存储成本？

- **链路追踪**：跨服务的请求如何追踪？

### 盲点 5：团队与组织

三个方案都是技术视角，忽略了：

- **人员招聘**：需要什么样的工程师？薪资预算？

**关于配置爆炸**：
- 并非每个容错点都需要独立配置。我们采用**分层配置策略**：
  - 全局默认值：熔断阈值 50%，超时 3s，重试 3 次
  - 服务级覆盖：仅关键服务（LLM、ASR）自定义参数
  - 实际需要调优的配置点约 4-5 个，非 10 个

**关于级联失败**：
- 采用**熔断器优先级机制**：下游熔断时，上游不再重试
- 设计**全局降级开关**：检测到雪崩前兆（错误率 > 30% + 延迟 > 1s），自动进入降级模式
- 引入**舱壁隔离**：不同服务的线程池隔离，防止单点耗尽资源

**关于熔断恢复**：
- Half-Open 状态下，限制试探请求为 1 个，避免反复熔断
- 引入**指数退避恢复**：Open → Half-Open 间隔从 30s 逐步延长到 5min
- 监控面板实时展示熔断状态，运维可人工干预

**方法论**：配置参数通过灰度发布和 A/B 测试逐步调优，而非一次性定死。

**短期修正（MVP 阶段）**：
1. 简化限流策略：WebSocket 独立通道，HTTP API 限流

### 对Q3的回应

**缓存失效流程：**

```
L1 失效（本地缓存）：
  1. 写操作先更新 DB
  2. 通过 Redis Pub/Sub 广播失效消息
  3. 所有节点收到消息，删除本地缓存
  4. 延迟双删：500ms 后再次删除（防止主从延迟）

同步延迟：通常 < 10ms
极端情况：节点网络隔离 → 本地缓存过期（TTL 5s）自动刷新
```

**缓存穿透防护：**

```
方案：布隆过滤器 + 空值缓存
  - 启动时加载所有有效 ID 到布隆过滤器
  - 布隆过滤器判断不存在 → 直接返回
  - 查询 DB 返回空 → 缓存空值（TTL 30s）
```

**缓存雪崩防护：**

```
方案：过期时间随机化
  - 基础 TTL：5 分钟
  - 随机偏移：0-60 秒
  - 热点数据：永不过期 + 主动刷新
```

**缓存击穿防护：**

```
方案：分布式锁 + 单飞模式
  - Key 过期时，第一个请求获取分布式锁
  - 其他请求等待，共享第一个请求的结果
  - 实现：Redis SETNX 或 singleflight.Group
```

### 对Q5的回应

  - 切换后：无感知（对话继续）

```

**非法内容处理：**

```

  1. 前端上传 → 临时存 MinIO（隔离桶）
  2. 调用阿里云内容审核 API
  3. 审核通过 → 移至正式桶
  4. 审核失败 → 删除临时文件，返回错误

**数据库删除：**

### 承认的不足

1. **Go + Rust 混合架构**：团队需要同时掌握两种语言，招聘和维护成本确实增加。纯 Go 团队难以落地。

2. **压测数据缺失**：单机 50k 连接是理论推算，无真实压测数据支撑。

3. **成本模型不完善**：未考虑用户增长后的成本曲线，缺乏成本预警机制。

4. **数据安全与合规**：原方案未涉及数据留存、导出、匿名化策略。

5. **可观测性不足**：缺少链路追踪，跨服务问题定位困难。
