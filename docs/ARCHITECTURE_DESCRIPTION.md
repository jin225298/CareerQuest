# 面试系统架构说明

## 一、系统概述

面试修炼手册是一个面向求职者的 AI 面试练习平台。系统采用前后端分离架构，前端使用 React + TypeScript，后端使用 FastAPI + SQLite，AI 能力由火山引擎 Doubao 模型提供。核心功能包括 AI 面试练习、智能记忆系统、个性化头像生成、社交互动等。

## 二、技术架构

### 2.1 前端架构

前端采用单页应用（SPA）模式，使用 React 18 作为核心框架，配合 React Router 实现客户端路由。状态管理采用 React Hooks（useState、useEffect）结合 React Query 进行服务端状态缓存。UI 组件使用 TailwindCSS 进行样式设计，Framer Motion 提供动画效果。

前端主要页面包括：主页（面试入口）、面试页面（文字/语音模式）、结果页面、历史记录、成就系统、好友系统、打卡系统、简历设计器、情感树洞、AI 导师对话等。每个页面独立成文件，通过路由配置进行管理。

### 2.2 后端架构

后端采用 FastAPI 框架，提供 RESTful API 接口。数据持久化使用 SQLite（通过 SQLAlchemy ORM），支持异步操作。服务层按功能模块划分，包括面试服务、用户服务、记忆服务、头像服务、好友服务等。

后端核心服务包括：
- 面试服务：管理面试会话、对话处理、评分生成
- 记忆服务：用户弱点的智能记忆、分层存储、动态检索
- 头像服务：像素风格头像生成、情绪动画管理
- 报告服务：面试报告生成、维度分析

### 2.3 AI 能力集成

系统接入火山引擎的 Doubao 系列模型：
- 对话模型（doubao-1-5-pro）：用于面试对话、问题生成、评估反馈
- 图像生成模型（doubao-seedream）：用于像素头像生成、情绪动画帧生成

## 三、面试核心流程

### 3.1 面试启动

用户在主页点击 FIGHT 按钮后，系统展示面试选择面板。用户可选择文字面试或语音面试模式，并设置面试类型（HR 面试/技术面试/行为面试）和难度等级。

当用户确认开始面试时，前端调用 POST /interviews/start 接口。后端执行以下操作：

首先，生成唯一的会话标识（session_id），用于追踪整个面试过程。然后，调用记忆服务获取该用户的历史记忆，包括必考项（L1 层）、参考项（L2 层）和抽查项（L3 层）。这些记忆反映了用户过往的技术弱点和常见错误。

接下来，系统将用户记忆与基础面试提示词合并，构建增强版的系统提示。例如，如果用户在 Redis 跳表实现上屡次犯错，系统会在提示词中强调："你需要重点关注候选人对 Redis 核心数据结构的理解，特别是跳表实现原理。"

最后，AI 模型基于增强提示生成第一个面试问题，返回给前端展示。会话信息同时持久化到数据库。

### 3.2 面试对话

面试对话采用回合制模式。用户回答问题后，前端调用 POST /interviews/reply 接口，传递用户回答内容。

后端接收回答后，将其追加到会话历史中。AI 模型基于当前对话上下文，决定下一步行为：如果回答不完整，进行追问以深入了解；如果回答充分，提出下一个相关问题；如果发现知识盲点，进行针对性考察。

AI 回复通过流式方式返回，确保用户能实时看到回答生成过程，提升交互体验。对话历史实时更新，保持在合理长度内（最近 10 轮对话），避免上下文过长。

### 3.3 面试结束

用户主动结束面试或达到最大对话轮数时，系统触发面试结束流程。前端调用 POST /interviews/end 接口。

后端执行一系列收尾操作：

评分生成：AI 模型基于完整对话历史，对候选人进行综合评估。评分维度包括回答准确性、表达能力、逻辑思维等，最终输出 0-100 分的分数和简短反馈。

心情值更新：根据面试分数计算心情值变化。高分（>85）增加 10-20 点心情值，低分（<60）减少 10-20 点。心情值影响用户在主页显示的头像情绪。

成就检查：检查本次面试是否触发成就解锁，如"首次面试"、"连续面试 7 天"、"高分达人"等。

报告生成（异步）：生成详细的面试报告，包括各维度得分、问题分析、改进建议等。报告生成耗时较长，通过后台任务异步执行。

记忆处理（异步）：分析本次面试对话，识别用户的技术弱点和知识盲点，更新记忆库。如果是重复出现的错误，将记忆升级到更高优先级；如果是已纠正的问题，将记忆降级。

## 四、智能记忆系统

### 4.1 记忆分层

系统将用户记忆分为三个层级，反映不同的重要程度：

L1 层（必考项）：用户多次犯错且未纠正的问题，属于顽固性弱点。系统在每次面试中都会重点考察这些内容。例如，某用户在 Redis 跳表实现上连续 3 次回答错误，该问题将被标记为 L1。

L2 层（参考项）：用户近期首次犯错的领域，属于潜在弱点。系统会选择性追问，观察用户是否已改进。

L3 层（抽查项）：用户曾经犯错但已纠正的问题。系统偶尔抽查，验证用户是否真正掌握，避免死记硬背。

### 4.2 记忆生命周期

记忆的创建发生在面试结束时。系统分析用户回答，识别错误或不足之处，通过脱敏处理移除敏感信息（姓名、公司名等），然后生成结构化的记忆摘要。

记忆的升级遵循"重复触发"原则。当新错误与已有记忆高度相似（向量相似度 > 0.85）时，系统增加该记忆的错误计数。当错误计数达到阈值（默认 2 次），记忆从 L2 升级到 L1。

记忆的降级遵循"正确回答"原则。当用户在后续面试中正确回答某知识点的问题时，系统增加该记忆的正确计数。正确计数达到阈值后，记忆降级到 L3。

### 4.3 记忆检索

在面试开始时，系统从记忆库中检索用户记忆，用于构建增强提示：

L1 记忆全部加载，确保必考项得到重点关注。L2 和 L3 记忆按比例抽样，避免提示过长影响 AI 生成质量。默认配置下，L2 抽取 30%（最少 3 条，最多 10 条），L3 抽取 10%（最少 1 条，最多 3 条）。

## 五、数据模型

### 5.1 用户模型

用户表（users）存储核心用户信息：用户 ID、昵称、心情值、战力值、胜场数、求职画像（职业方向、工作经验、目标岗位等）、头像路径（三种情绪的 GIF 和 PNG）、当前情绪状态。

### 5.2 面试模型

面试表（interviews）记录每次面试会话：会话 ID、用户 ID、面试类型、难度、目标岗位、对话历史、评分、时长、关联报告 ID、创建时间、完成时间。

面试报告表（interview_reports）存储详细报告：报告 ID、会话 ID、综合分数、等级（S/A/B/C）、各维度得分、问题分析、改进建议、元数据。

### 5.3 记忆模型

记忆元数据表（memory_meta）存储记忆主体：记忆 ID、用户 ID、层级（L1/L2/L3）、错误类型、主题、摘要、错误次数、正确次数、向量哈希、来源会话、创建时间、升级时间、降级时间。

记忆事件表（memory_events）记录状态变化日志：记忆 ID、事件类型（创建/升级/降级）、原层级、新层级、触发原因、会话 ID、时间戳。

## 六、头像系统

### 6.1 头像生成

用户上传照片后，系统调用 AI 图像生成模型，生成像素风格的头像动画。生成流程如下：

首先，上传参考图（像素风格示例）和用户照片到火山引擎 API。然后，针对每种情绪（开心/难过/激动），生成 8 帧连续动画帧。接着，对每帧进行绿幕抠图处理，移除绿色背景。最后，将帧序列合成为 GIF 动画，并保存首帧为静态 PNG。

生成的三种情绪头像分别存储为 6 个文件（3 个 GIF + 3 个 PNG），路径保存到用户表的对应字段。

### 6.2 情绪切换

头像的情绪展示由心情值驱动。系统定义映射规则：心情值 < 30 显示难过头像，30-70 显示开心头像，> 70 显示激动头像。

当用户进入主页时，系统查询当前心情值，自动加载对应情绪的头像。用户也可以手动切换情绪，系统记录用户选择但不改变心情值。

心情值的变化由多种事件触发：面试成绩、完成任务、社交互动、每日登录等。系统维护心情值在 0-100 范围内，并设置保底机制（不低于 30），防止用户流失。

## 七、扩展功能

### 7.1 好友系统

用户可以添加好友、查看好友动态、发起面试邀请。好友之间可以比较面试成绩、共享学习进度。好友请求需要对方确认，支持删除好友操作。

### 7.2 打卡系统

每日打卡系统鼓励用户持续练习。用户完成打卡后获得奖励（战力值、心情值提升）。系统记录连续打卡天数，支持打卡日历展示和排行榜功能。

### 7.3 成就系统

成就系统通过勋章奖励激励用户。成就类型包括：里程碑成就（首次面试、累计面试数）、技能成就（高分达人、连胜王者）、社交成就（交友广泛）等。解锁成就时弹出动画提示，并更新用户档案。

### 7.4 简历设计器

简历设计器帮助用户创建专业简历。用户填写基本信息、教育背景、工作经历、项目经验、技能特长后，系统生成格式化的简历预览，支持导出功能。

### 7.5 情感树洞

情感树洞提供匿名分享空间。用户可以发布求职过程中的心情、困惑、经验，其他用户可以点赞和评论。树洞内容脱敏处理，保护用户隐私。

### 7.6 AI 导师对话

AI 导师功能提供个性化指导。用户可以与 AI 导师进行自由对话，咨询求职问题、技术难题、职业规划等。导师会结合用户的求职画像和历史表现，给出针对性建议。

## 八、部署架构

系统采用单机部署模式，适合中小规模用户群体。前端静态资源部署在 Nginx 或 CDN 上，后端服务运行在 Python 虚拟环境中，数据库使用 SQLite 文件数据库，AI 能力通过火山引擎云 API 调用。

如需扩展到大规模用户，可进行以下优化：将 SQLite 替换为 PostgreSQL 或 MySQL，引入 Redis 缓存热点数据，使用 Celery 处理异步任务，部署多实例负载均衡。
