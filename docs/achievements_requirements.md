## 一、需求解构

### 1.1 核心目标

| 目标层级 | 描述 |
|---------|------|
| **业务目标** | 提升用户留存率、日活时长、面试完成率 |
| **产品目标** | 构建正向反馈循环，增强用户成就感与归属感 |
| **用户目标** | 记录成长轨迹，获得阶段性激励，感知进步可视化 |

### 1.2 目标用户价值

```
┌─────────────────────────────────────────────────────────────┐
│                      用户价值金字塔                           │
├─────────────────────────────────────────────────────────────┤
│  自我实现层  │ 成就展示 → 社交货币 → 个人品牌建设            │
├─────────────────────────────────────────────────────────────┤
│  尊重层      │ 成就解锁 → 能力认可 → 自信提升                │
├─────────────────────────────────────────────────────────────┤
│  社交层      │ 成就分享 → 话题引子 → 同类识别                │
├─────────────────────────────────────────────────────────────┤
│  安全层      │ 进度可见 → 成长可量化 → 焦虑缓解              │
├─────────────────────────────────────────────────────────────┤
│  生理层      │ 基础激励 → 行为引导 → 习惯养成                │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 使用场景

| 场景 | 用户行为 | 成就触发 | 用户心理 |
|------|---------|---------|---------|
| **新手引导** | 完成首次面试 | `first_interview` | "我入门了" |
| **日常练习** | 连续多日登录 | `weekly_login` | "我坚持下来了" |
| **能力验证** | 获得高分评价 | `perfect_score` | "我很厉害" |
| **挑战自我** | 尝试长对话面试 | `marathon` | "我突破了" |
| **效率追求** | 快速完成面试 | `quick_finish` | "我很高效" |

## 二、可行性预判

### 2.1 技术实现难度评估

| 成就代码 | 实现难度 | 技术要点 | 工时估算 |
|---------|---------|---------|---------|
| `first_interview` | ⭐ 低 | 面试完成时计数判断 | 0.5h |
| `interviewer_10` | ⭐ 低 | 累计计数，已有数据支撑 | 0.5h |
| `interviewer_50` | ⭐ 低 | 同上 | 0.5h |
| `perfect_score` | ⭐ 低 | 评分系统已存在 | 0.5h |
| `streak_3` | ⭐⭐ 中 | 需新增连续性追踪逻辑 | 2h |
| `streak_5` | ⭐⭐ 中 | 同上 | 0.5h |
| `quick_finish` | ⭐ 低 | 时长计算，已有数据 | 1h |
| `marathon` | ⭐ 低 | 对话轮次统计 | 1h |
| `low_stress` | ⭐⭐ 中 | 需追踪全程压力值变化 | 2h |
| `weekly_login` | ⭐⭐⭐ 高 | 需新增登录记录表 | 4h |

**总工时估算**：12-15h（后端 + 前端 + 测试）

### 2.2 数据依赖分析

```
已有数据 ✓
├── interview_sessions（面试记录）
│   ├── id, user_id, created_at
│   ├── score（评分）
│   ├── duration（时长）
│   └── round_count（对话轮次）——需确认是否存在
│
├── stress_records（压力记录）——需确认结构
│   └── 当前仅有实时压力值？是否有历史记录？
│
└── user_stats（用户统计）——需确认是否存在
    └── total_interviews, last_login_date 等

需要新增 ✗
├── user_achievements（成就记录表）
│   ├── id, user_id, achievement_code
│   ├── unlocked_at, notified_at
│   └── metadata（JSON，扩展字段）
│
├── login_streaks（连续登录记录）
│   ├── user_id, current_streak, longest_streak
│   └── last_login_date
│
└── interview_streaks（连续高分记录）
    ├── user_id, current_streak
    └── last_interview_id
```

### 2.3 风险识别

| 风险类型 | 风险描述 | 概率 | 影响 | 应对策略 |
|---------|---------|------|------|---------|
| **数据缺失** | `round_count` 字段可能不存在 | 中 | 高 | 需先确认数据结构，必要时迁移补充 |
| **压力记录** | 无历史压力数据，无法追溯 | 中 | 中 | 仅对新面试生效，或从 now 开始记录 |
| **刷成就** | 用户可能通过短面试刷量 | 高 | 中 | 增加面试有效性校验（如最低轮次） |
| **成就通知** | 解锁时机与通知体验 | 中 | 中 | 设计非侵入式通知（Toast/弹窗） |
| **历史数据** | 老用户是否补发成就 | 低 | 高 | 需明确策略：是否回溯计算 |

## 三、创新性建议

### 3.1 成就类型扩展建议

| 建议成就 | 类型 | 触发条件 | 价值点 |
|---------|------|---------|--------|
| `all_achievements` | 集齐型 | 解锁全部成就 | 终极目标，长期留存 |
| `night_owl` | 时间型 | 23:00-6:00 完成面试 | 趣味性，用户画像 |
| `comeback` | 回归型 | 停留7天后回归 | 挽留流失用户 |
| `first_share` | 社交型 | 首次分享成就 | 病毒传播 |
| `challenge_master` | 挑战型 | 完成所有难度面试 | 体现平台深度 |

### 3.2 用户激励优化建议

```
当前设计                    建议优化
───────────────────────────────────────────────
仅成就解锁        →    成就 + 奖励组合
                        ├── 解锁特效/动画
                        ├── 虚拟道具（如新头像框）
                        └── 经验值/积分加成

单一展示          →    分层展示体系
                        ├── 个人主页成就墙
                        ├── 成就详情页（解锁时间、进度）
                        └── 成就分享卡片（社交传播）

一次性解锁        →    进度可视化
                        ├── 进度条（如 3/10 次面试）
                        ├── 即将解锁提示
                        └── 多级成就（铜→银→金）
```

### 3.3 优先级标注

| 优先级 | 成就代码 | 理由 |
|--------|---------|------|
| **P0 必做** | `first_interview` | 新手留存关键 |
| **P0 必做** | `interviewer_10` | 核心行为激励 |
| **P0 必做** | `perfect_score` | 能力认可，强激励 |
| **P1 重要** | `streak_3`, `streak_5` | 培养习惯 |
| **P1 重要** | `weekly_login` | 提升日活 |
| **P2 可选** | `quick_finish`, `marathon` | 趣味性成就 |
| **P2 可选** | `low_stress` | 数据依赖较高 |

## 四、确认清单

### 4.1 数据结构确认

- [ ] `interview_sessions` 表是否包含 `round_count`（对话轮次）字段？
- [ ] 压力值是否有历史记录？还是仅实时计算？
- [ ] 是否有 `user_stats` 汇总表？还是需要新建？
- [ ] 评分 `score` 的满分是多少？100分制？

### 4.2 业务逻辑确认

- [ ] **历史数据处理**：老用户的面试次数是否追溯计算成就？
- [ ] **重复解锁**：同一成就是否可多次解锁（如多级成就）？
- [ ] **面试有效性**：是否有最低有效性要求（如最少3轮对话）？
- [ ] **连续性定义**：`streak_3` 的"连续"指什么？连续3次面试？还是连续3天？
- [ ] **登录定义**：`weekly_login` 是否允许中断？还是必须连续7天？

### 4.3 产品体验确认

- [ ] **通知方式**：成就解锁时如何通知用户？（Toast/弹窗/Push）
- [ ] **展示位置**：成就展示在哪里？（个人主页/独立页面/面试结束页）
- [ ] **分享功能**：是否支持成就分享？分享到哪些平台？
- [ ] **成就图标**：是否已有设计稿？还是需要设计资源？

### 4.4 技术实现确认

- [ ] **触发时机**：面试结束时实时计算？还是异步任务？
- [ ] **性能考量**：是否需要缓存用户成就状态？
- [ ] **扩展性**：未来是否支持自定义成就？动态配置？
