## 1. 设计理念

### 1.1 三维平衡原则

```
                    性能
                     ▲
                    /|\
                   / | \
                  /  |  \
    开发效率 <----+--+----> 安全
                  \  |  /
                   \ | /
                    \|/
                     ▼
                   用户体验
```

**核心理念**：
- **性能是底线**：面试体验流畅是产品价值的前提
- **安全是边界**：用户数据保护决定产品能走多远
- **效率是节奏**：快速验证 + 渐进式优化，避免过度设计

### 1.2 分阶段投入策略

| 阶段 | 性能投入 | 安全投入 | 开发效率 | 目标 |
|------|----------|----------|----------|------|
| MVP (W1-W4) | 基础优化 | 核心防护 | 最高优先 | 验证产品假设 |
| V1.0 (W5-W8) | 性能调优 | 完善防护 | 中等优先 | 提升用户体验 |
| V1.5 (W9-W12) | 极致优化 | 安全加固 | 按需投入 | 差异化竞争 |
| V2.0 (W13+) | 规模化 | 合规认证 | 持续迭代 | 商业化运营 |

### 1.3 关键决策

| 决策点 | 选择 | 来源 | 理由 |
|--------|------|------|------|
| 后端语言 | Go (纯Go，无Rust) | A修正 | 降低团队门槛，避免FFI复杂度 |
| 架构风格 | 分层架构 → 渐进式DDD | B修正 | MVP不过度设计，按需演进 |
| 认证方案 | JWT (自建，可选Supabase) | C修正 | MVP可用Supabase快速验证，后期可迁移 |
| 数据库 | PostgreSQL + Redis | A+B | 成熟稳定，读写分离支持扩展 |
| LLM服务商 | 通义千问主 + 文心一言备 | 需求确认 | 国内合规，成本可控 |
| 像素生成 | Seedance 5.0 | 需求确认 | 已验证可行 |

## 2. 演化路径（来源追溯表）

### 2.1 架构特性来源

| 特性 | 来源 | 具体采纳 | 修正说明 |
|------|------|----------|----------|
| 多级缓存 | 方案A | L1本地 + L2 Redis + L3 DB | 补充缓存失效策略和穿透防护 |
| 音频流式处理 | 方案A | 100ms切片 + WebSocket | 补充弱网应对策略 |
| 熔断降级 | 方案B | 分级降级(L1/L2/L3) | 补充预录视频面试库 |
| 安全防护 | 方案B | 全链路安全 | 已评估性能影响(+7ms) |
| MVP快速验证 | 方案C | 4周核心功能 | 补充技术债务偿还计划 |
| 成熟服务复用 | 方案C | Supabase/托管服务 | 明确迁移路径 |

### 2.2 辩论修正点

| 辩论问题 | 原方案问题 | 修正方案 | 影响 |
|----------|------------|----------|------|
| Go+Rust混合 | 团队门槛高，FFI复杂 | 纯Go，热点优化用独立服务 | 开发效率↑，性能损失15% |
| 限流策略 | 60req/min与面试冲突 | WebSocket独立通道，HTTP API单独限流 | 避免误杀 |
| 降级策略 | "预设回复"体验差 | 分级降级 + 预录视频库(50题) | 降级仍可完成面试 |
| DDD复杂度 | MVP阶段过重 | 分层架构起步，渐进式引入 | 降低学习成本 |
| Supabase锁定 | 无退出策略 | 明确迁移成本(6-10人天) | 风险可控 |
| 技术债务 | 无偿还计划 | 按DAU阈值触发基建投入 | 渐进式投入 |

```typescript
// 客户端实时计算，延迟<10ms
const LipSyncAlgo = {
  calculateVolume(audioData: Float32Array): number {
    let sum = 0;
    for (let i = 0; i < audioData.length; i++) {
      sum += audioData[i] * audioData[i];
    }
    return Math.sqrt(sum / audioData.length);
  },
  
  volumeToFrame(volume: number): number {
    return Math.floor(volume * 5); // 0-5 帧索引
  },
  
  animate() {
    this.analyser.getFloatTimeDomainData(this.dataArray);
    const volume = this.calculateVolume(this.dataArray);
    const frame = this.volumeToFrame(volume);
    this.updateMouthFrame(frame);
    requestAnimationFrame(() => this.animate());
  }
```

## 8. 风险与缓解

### 8.1 技术风险

| 风险 | 级别 | 影响 | 缓解措施 | 来源 |
|------|------|------|----------|------|
| LLM首字延迟不稳定 | 高 | 面试体验差 | 流式输出 + 预测预加载 + 对话缓存 | A方案 |
| 第三方AI服务不稳定 | 高 | 核心功能不可用 | 多供应商 + 预录视频面试库 | B方案修正 |
| WebSocket连接数突增 | 高 | 服务崩溃 | 限流 + 弹性伸缩 + 压测验证 | A方案 |
| 缓存雪崩 | 高 | 数据库压力暴增 | TTL随机化 + 熔断 + 多级缓存 | A+B方案 |
| Lip Sync延迟 | 中 | 动画不同步 | 本地Web Audio API处理 | A方案 |
| 弱网环境音频丢失 | 中 | 识别不准确 | 切片编号 + 重传机制 | 辩论修正 |

### 8.2 成本风险

| 风险 | 级别 | 影响 | 缓解措施 |
|------|------|------|----------|
| AI API调用成本高 | 高 | 利润率低 | 对话缓存(省30%) + 监控预警 |
| 用户增长成本线性增长 | 高 | 不可持续 | 按DAU阈值触发优化 + 批量调用 |
| CDN流量成本 | 中 | 成本增加 | 缓存策略优化 + WebP/AVIF |

**成本预测模型（辩论补充）**:

| 用户量 | 月成本(估算) | 说明 |
|--------|-------------|------|
| 500 | ¥3,000 | MVP验证期 |
| 5,000 | ¥25,000 | 需优化策略 |
| 50,000 | ¥200,000+ | 需架构重构 |

### 8.3 产品风险

| 风险 | 级别 | 影响 | 缓解措施 |
|------|------|------|----------|
| 用户学习成本高 | 中 | 流失率高 | 新手引导、渐进式功能解锁 |

## 9. 已知权衡

### 9.1 架构权衡决策记录（ADR）

| ID | 决策 | 选择方案 | 放弃方案 | 理由 |
|----|------|---------|---------|------|
| ADR-001 | 后端语言 | Go (纯Go) | Go+Rust混合 | 降低团队门槛，避免FFI复杂度 |
| ADR-002 | 架构风格 | 分层架构 | 完整DDD | MVP不过度设计，渐进式演进 |
| ADR-003 | 认证方案 | JWT (可选Supabase) | Session | 无状态、易扩展、可迁移 |
| ADR-004 | 数据库 | PostgreSQL | MySQL | 行级安全、透明加密更完善 |
| ADR-005 | 缓存 | Redis Cluster | 单机Redis | 高可用、分片、故障转移 |
| ADR-006 | 消息队列 | NATS JetStream | Kafka | 轻量级、运维成本低 |
| ADR-007 | 容器编排 | Kubernetes | Docker Swarm | 生态成熟、安全特性完善 |
| ADR-008 | API风格 | REST + WebSocket | GraphQL | 安全边界清晰、缓存友好 |
| ADR-009 | 限流策略 | HTTP/WebSocket分离 | 统一限流 | 避免面试场景误杀 |

### 9.2 性能vs安全权衡

| 场景 | 选择 | 影响 | 接受度 |
|------|------|------|--------|
| 数据传输 | 全程HTTPS + 请求签名 | 延迟增加5-10ms | 可接受 |
| 数据存储 | 敏感数据加密存储 | 读写性能降低10-20% | 可接受 |
| 外部调用 | 熔断+重试+超时 | 最坏情况延迟增加3倍 | 可接受 |
| API响应 | 全字段校验 | 每请求增加1-3ms | 可接受 |

**结论**: 安全防护带来的性能损失（约7ms增量）相比LLM响应时间（1-3s）可忽略不计。

### 9.3 功能vs稳定权衡

| 场景 | 选择 | 影响 |
|------|------|------|
| 新功能发布 | 灰度发布+功能开关 | 上线周期延长 |
| 外部服务依赖 | 多服务商备份+降级 | 成本增加 |
| 数据一致性 | 最终一致性 | 用户可能看到旧数据 |

### 9.4 开发效率vs可维护性权衡

| 场景 | 选择 | 影响 |
|------|------|------|
| MVP阶段技术债务 | 接受，按DAU阈值偿还 | 前期开发快 |
| 代码规范 | 渐进式引入 | 前期灵活，后期规范 |
| 测试要求 | MVP最低限度，逐步提高 | MVP快速上线 |

## 附录C：故障演练脚本

### C.1 LLM服务故障演练

```yaml
场景: 通义千问完全不可用 (持续30分钟)
触发: API返回503, 连续5次失败
动作:
  1. 熔断器打开 (30s内)
  2. 自动切换文心一言
  3. 显示"正在切换服务"提示
  4. 切换成功后继续面试

验证:
  - 切换时间 < 30s
  - 对话上下文保留
  - 用户无感知中断
```

### C.2 数据库故障演练

```yaml
场景: PostgreSQL主库不可用
触发: 连接超时, 写入失败
动作:
  1. 检测主库故障 (10s内)
  2. 只读模式运行
  3. 从库提升为主库
  4. 恢复写入

验证:
  - RPO: 最多丢失4小时
  - RTO: < 30分钟
  - 只读期间用户可继续面试
```
